# سياسة النسخ الاحتياطي — Backup Policy

**المشروع:** Qatra - منصة التبرع بالدم في الجزائر
**الإصدار:** 1.0.0
**آخر تحديث:** 2026-07-13
**المسؤول:** فريق التطوير + DPO

---

## 1. المقدمة

تحدد هذه السياسة إجراءات النسخ الاحتياطي واستعادة البيانات لمنصة Qatra، وذلك لضمان استمرارية الخدمة والحماية من فقدان البيانات.

---

## 2. المقاييس الأساسية — Key Metrics

| المقياس | القيمة | الوصف |
|---------|--------|-------|
| **RPO (Recovery Point Objective)** | 24 ساعة | أقصى مدة مقبول لفقدان البيانات |
| **RTO (Recovery Time Objective)** | 4 ساعات | أقصى مدة مقبولة لاستعادة الخدمة |
| **مدة الاحتفاظ اليومية** | 30 يوماً | عدد النسخ الاحتياطية اليومية المحفوظة |
| **مدة الاحتفاظ الشهرية** | 12 شهراً | عدد النسخ الاحتياطية الشهرية المحفوظة |
| **الشفافية الجغرافية** | مُدارة من Supabase | النسخ الاحتياطية موزعة جغرافياً عبر مراكز بيانات Supabase |

---

## 3. أنواع النسخ الاحتياطية

### 3.1 النسخ الاحتياطية التلقائية (Supabase Managed)

| العنصر | التفاصيل |
|--------|----------|
| **النوع** | Daily automated backups |
| **الكثافة** | يومياً |
| **المدة** | 30 يوماً (آخر 30 نسخة يومية) |
| **التخزين** | Supabase managed storage (مشفر) |
| **الطريقة** | WAL (Write-Ahead Logging) + base backup |
| **الشفافية** | مُدارة بالكامل من Supabase |
| **التشفير** | AES-256 at rest |

### 3.2 النسخ الاحتياطية الشهرية

| العنصر | التفاصيل |
|--------|----------|
| **النوع** | Monthly full backups |
| **الكثافة** | شهرياً |
| **المدة** | 12 شهراً |
| **التخزين** | Supabase managed storage |
| **الطريقة** | Full database dump |

### 3.3 النسخ الاحتياطية اليدوية (SQL Exports)

| العنصر | التفاصيل |
|--------|----------|
| **النوع** | Manual SQL exports |
| **الكثافة** | عند التحديثات الكبيرة أو قبل النشر |
| **المدة** | 90 يوماً |
| **التخزين** | مخزن مشفر خارجي |
| **الطريقة** | pg_dump أو Supabase Dashboard |

## 4. ما يُخزَّن في النسخة الاحتياطية

### 4.1 البيانات المشمولة

```
✅ جدول المتبرعين (donors) — البيانات الشخصية المشفرة
✅ جدول طلبات الدم (blood_requests)
✅ جدول التقييمات (ratings)
✅ جدول الملاحظات (feedback)
✅ جداول المرجعية (wilayas, communes, blood_types)
✅ جدول سجلات الموافقات (consent_records)
✅ جدول سجلات التدقيق (audit_logs)
✅ Schema والمُحدّثات (migrations)
✅ الدوال والآليات المحفوظة (stored procedures)
```

### 4.2 ما لا يُخزَّن

```
❌ ملفات المستخدمين المرفوعة (إن وُجدت)
❌ بيانات الجلسة (session data)
❌ بيانات Cache
❌ المتغيرات البيئية (Environment Variables)
```

## 5. التشفير — Encryption

### 5.1 التشفير أثناء الراحة (At Rest)

| العنصر | التفاصيل |
|--------|----------|
| **الخوارزمية** | AES-256 |
| **الإدارة** | Supabase managed |
| **المفاتيح** | تُدار تلقائياً من Supabase |
| **التوثيق** | موثق في Supabase security documentation |

### 5.2 التشفير أثناء النقل (In Transit)

| العنصر | التفاصيل |
|--------|----------|
| **البروتوكول** | TLS 1.2+ |
| **الاستخدام** | جميع اتصالات قاعدة البيانات |
| **شهادة SSL** | مُدارة من Supabase |

### 5.3 ملاحظات أمنية

- لا تُخزَّن مفاتيح التشفير يدوياً
- مفاتيح Supabase تُدار تلقائياً
- النسخ الاحتياطية مشفرة حتى أثناء النقل من Supabase

## 6. اختبار الاستعادة — Recovery Testing

### 6.1 جدول الاختبارات

| التكرار | الإجراء | المسؤول |
|---------|---------|--------|
| **شهرياً** | اختبار استعادة كامل على بيئة تطوير | المطور |
| **ربع سنوياً** | اختبار استعادة وقياس RTO الفعلي | فريق التطوير |
| **سنوياً** | اختبار كارثة كامل | الفريق الكامل |

### 6.2 إجراءات اختبار الاستعادة

```
1. إنشاء بيئة تطوير مؤقتة
2. استعادة أحدث نسخة احتياطية
3. التحقق من سلامة البيانات:
   - عدد السجلات في كل جدول
   - التحقق من المراجع (foreign keys)
   - التحقق من البيانات المشفرة
4. تشغيل اختبارات التطبيق على البيانات المستعاد
5. قياس الوقت المستغرق (RTO)
6. توثيق النتائج
7. تدمير البيئة المؤقتة
```

### 6.3 معايير النجاح

| المعيار | القيمة المطلوبة |
|---------|----------------|
| RTO الفعلي | ≤ 4 ساعات |
| سلامة البيانات | 100% |
| عدد السجلات | مطابق للنسخة الأصلية |
| المراجع | سليمة بالكامل |

## 7. إعادة التشغيل — Recovery Procedures

### 7.1 سيناريو 1: فقدان جزئي للبيانات

```
1. تحديد النطاق المتضرر
2. تحديد آخر نسخة احتياطية سليمة
3. استعادة البيانات من النسخة الاحتياطية
4. التحقق من السلاسة
5. استئناف الخدمة
6. تقرير الحادث
```

### 7.2 سيناريو 2: فقدان كامل لقاعدة البيانات

```
1. إنشاء قاعدة بيانات جديدة على Supabase
2. استعادة أحدث نسخة احتياطية كاملة
3. تطبيق أي WAL logs متبقية
4. التحقق من سلامة البيانات
5. تحديث متغيرات البيئية في التطبيق
6. اختبار التطبيق بالكامل
7. توجيه حركة المرور إلى القاعدة الجديدة
8. تأكيد الاستقرار
9. تقرير مفصل
```

### 7.3 سيناريو 3: حذف غير متوقع لجدول واحد

```
1. تحديد الجدول المتضرر
2. استعادة الجدول من آخر نسخة احتياطية
3. التحقق من المراجع (foreign keys)
4. التحقق من سلامة البيانات المرتبطة
5. استئناف الخدمة
```

## 8. النسخ الاحتياطي اليدوي — Manual Backup

### 8.1 متطلبات النسخ الاحتياطي اليدوي

يجب عمل نسخة احتياطية يدوية قبل:
- تحديثات Schema الكبيرة
- تغييرات معمارية كبيرة
- نسخة إصدار جديدة (release)
- أي عملية قد تؤثر على البيانات

### 8.2 كيفية عمل نسخة احتياطية يدوية

```bash
# عبر Supabase Dashboard
# Database → Backups → Create Backup

# عبر pg_dump (إذا توفرت صلاحية Direct Access)
pg_dump -h [HOST] -U [USER] -d [DATABASE] -F c -f backup_$(date +%Y%m%d).dump
```

### 8.3 تخزين النسخ اليدوية

- تُخزَّن في مخزن مشفر خارج Supabase
- تُسمَّى بالصيغة: `qatra_backup_YYYYMMDD_HHMMSS.dump`
- تُحذف بعد 90 يوماً

---

## 9. الشفافية الجغرافية — Geographic Redundancy

### 9.1 كيف يعمل Supabase

- تُخزَّن النسخ الاحتياطية تلقائياً في مراكز بيانات Supabase المتعددة
- تُدار الشفافية الجغرافية من قبل Supabase
- لا يمكن التحكم في الموقع الجغرافي يدوياً في الخطة المجانية

### 9.2 المخاطر والتحجيم

| المخاطرة | التحجيم |
|----------|---------|
| انقطاع مركز بيانات واحد | الشفافية التلقائية من Supabase |
| كارثة طبيعية | مراكز بيانات متعددة من Supabase |
| حادث أمني في مركز بيانات | تشفير البيانات at rest |

## 10. جدول الاحتفاظ التفصيلي

| نوع النسخة | الكثافة | المدة | عدد النسخ المحفوظة |
|------------|---------|-------|-------------------|
| يومية | يومياً | 30 يوماً | 30 |
| شهرية | شهرياً | 12 شهراً | 12 |
| يدوية | عند الحاجة | 90 يوماً | غير محدود |

### ملاحظات

- النسخ الاحتياطية التلقائية تتجاوز 30 يوماً تُحذف تلقائياً من Supabase
- النسخ الاحتياطية الشهرية تُحذف بعد 12 شهراً
- النسخ اليدوية تُحذف بعد 90 يوماً عبر سكربت
