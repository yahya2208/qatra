# خطة الاستجابة لحوادث أمنية — Security Incident Response Plan

**المشروع:** Qatra - منصة التبرع بالدم في الجزائر
**الإصدار:** 1.0.0
**آخر تحديث:** 2026-07-13
**المسؤول:** فريق الاستجابة للحوادث

---

## 1. تعريف الحادث الأمني

### 1.1 التعريف

**الحادث الأمني (Security Incident):** أي حدث يُهدد السرية (Confidentiality)، أو النزاهة (Integrity)، أو توفر (Availability) البيانات الشخصية أو النظام التقني للمنصة.

### 1.2 أمثلة

- وصول غير مصرح به لقاعدة البيانات
- تسريب بيانات شخصية للمتبرعين
- هجوم حقن SQL (SQL Injection)
- هجوم حجب الخدمة (DDoS)
- سرقة بيانات اعتماد المستخدمين
- تعديل غير مصرح به في البيانات
- فقدان نسخة احتياطية

---

## 2. فئات الحوادث والمستويات — Categories & Severity Levels

### 2.1 تصنيف الحوادث

| الفئة | الوصف |
|-------|-------|
| **F1: أمن البيانات** | تسريب، تعديل، أو تدمير بيانات شخصية |
| **F2: أمن التطبيق** | ثغرة في كود التطبيق (OWASP Top 10) |
| **F3: أمن البنية التحتية** | مشكلة في الخادم، قاعدة البيانات، أو الشبكة |
| **F4: أمن الحسابات** | اختراق حساب مستخدم أو مسؤول |
| **F5: الحوكمة** | خرق سياسات داخلية أو متطلبات قانونية |

### 2.2 مستويات الخطورة

| المستوى | الوصف | أمثلة | وقت الاستجابة |
|---------|-------|-------|---------------|
| **🔴 حرج (Critical)** | تسريب واسع النطاق لبيانات حساسة، أو توقف كامل للمنصة | تسريب بيانات المتبرعين جماعياً، اختراق قاعدة البيانات | **4 ساعات** |
| **🟠 مرتفع (High)** | تسريب محدود أو خرق أمني خطير | اختراق حساب مسؤول، ثغرة SQL injection مُستغلة | **24 ساعة** |
| **🟡 متوسط (Medium)** | محاولة خرق أو ثغرة غير مُستغلة | محاولة تسجيل دخول مشبوهة كثيرة، ثغرة غير معروفة | **72 ساعة** |
| **🟢 منخفض (Low)** | خرق بسيط أو مشكلة تقنية | خطأ في الإعدادات، محاولة فاشلة | **أسبوع** |

## 3. فريق الاستجابة — Incident Response Team

### 3.1 أعضاء الفريق

| الدور | المسؤولية | جهة الاتصال |
|-------|----------|-------------|
| **مسؤول حماية البيانات (DPO)** | الإشراف العام، التواصل مع CNPDP، التقييم القانوني | dpo@qatra.dz |
| **المطور الرئيسي (Lead Developer)** | التحقيق التقني، الاحتواء، الإصلاح | dev-lead@qatra.dz |
| **المحامي (Legal Counsel)** | التقييم القانوني، الإبلاغ للجهات المختصة | legal@qatra.dz |
| **مدير العمليات (Ops)** | مراقبة النظام، إدارة الخوادم | ops@qatra.dz |

### 3.2 سلسلة القيادة

```
الحادث → DPO (تنسيق) → الفريق حسب الدور → التقرير النهائي
```

## 4. مرحلة الاكتشاف — Detection Phase

### 4.1 مصادر الاكتشاف

| المصدر | التفاصيل |
|--------|----------|
| **Supabase Dashboard** | مراقبة الاستعلامات المشبوهة، معدل الطلبات |
| **Supabase Logs** | تحليل سجلات الوصول والأخطاء |
| **تنبيهات التطبيق** | أخطاء غير متوقعة، اختلافات في السلوك |
| **بلاغات المستخدمين** | تقارير من المتبرعين أو المرضى |
| **أدوات مراقبة خارجية** | Supabase monitoring tools |

### 4.2 إجراءات الاكتشاف

```
1. استقبال البلاغ أو اكتشاف التنبيه
2. التسجيل في سجل الحوادث (Incident Log)
3. التحقق من صحة البلاغ
4. تحديد مستوى الخطورة
5. تشكيل فريق الاستجابة
6. بدء مرحلة الاحتواء
```

### 4.3 تسجيل الحادث

يجب توثيق المعلومات التالية فوراً:
- التاريخ والوقت
- وصف الحادث
- المصدر (كيف تم اكتشافه)
- النطاق المتضرر
- مستوى الخطورة

## 5. مرحلة الاحتواء — Containment Phase

### 5.1 الاحتواء القصير المدى (فوراً)

| الإجراء | التفاصيل |
|---------|----------|
| **عزل النظام المتضرر** | تعطيل الوصول للنظام المتأثر |
| **تغيير كلمات المرور** | جميع الحسابات المعرضة للخطر |
| **حجب عناوين IP** | عناوين IP المشبوهة |
| **إيقاف الخدمات** | إيقاف مؤقت للخدمة المتأثر إن لزم |
| **توثيق** | تسجيل جميع الإجراءات المتخذة |

### 5.2 الاحتواء طويل المدى (خلال 24-72 ساعة)

| الإجراء | التفاصيل |
|---------|----------|
| **تحليل السبب الجذري** | كيف حدث الحادث؟ |
| **سد الثغرة** | إصلاح الثغرة الأمنية |
| **تحديث النظام** | تطبيق التحديثات الأمنية |
| **مراجعة الإعدادات** | تأكد من صحة جميع الإعدادات |
| **اختبار** | التأكد من نجاح الإصلاح |

### 5.3 أدوات الاحتواء

```bash
# تغيير مفاتيح Supabase (إن لزم)
# Supabase Dashboard → Settings → API → Rotate Keys

# حجب IP عبر Supabase (إن توفرت)
# Supabase Dashboard → Settings → Network Restrictions

# مراجعة RLS Policies
# Supabase Dashboard → Authentication → Policies
```

## 6. مرحلة التحقيق — Investigation Phase

### 6.1 الأسئلة الأساسية

| السؤال | التفاصيل |
|--------|----------|
| **ماذا حدث؟** | وصف دقيق للحادث |
| **متى حدث؟** | التوقيت الدقيق |
| **أين حدث؟** | النظام أو الخدمة المتضررة |
| **كيف حدث؟** | الطريقة أو الثغرة المستخدمة |
| **لماذا حدث؟** | السبب الجذري |
| **من تأثر؟** | المستخدمون أو البيانات المتضررة |
| **ما المدى؟** | حجم البيانات المتضررة |

### 6.2 التحقيق التقني

```
1. مراجعة سجلات Supabase (Logs)
2. تحليل سجلات التطبيق (Application Logs)
3. فحص سجلات المصادقة (Auth Logs)
4. تحليل استعلامات قاعدة البيانات المشبوهة
5. مراجعة RLS Policies
6. فحص إعدادات الأمان
7. تحليل Malware (إن وُجد)
```

### 6.3 التوثيق

- توثيق جميع الأدلة
- لقطات شاشة (Screenshots)
- نسخ من السجلات
- تقرير تقني مفصل

## 7. مرحلة الإصلاح — Remediation Phase

### 7.1 إجراءات الإصلاح

| الإجراء | المسؤول | المدة |
|---------|--------|-------|
| سد الثغرة الأمنية | المطور | فوراً |
| تحديث كلمات المرور | المسؤول | خلال 24 ساعة |
| تحديث RLS Policies | المطور | خلال 24 ساعة |
| تحديث المكتبات | المطور | خلال 48 ساعة |
| مراجعة جميع الإعدادات | الفريق | خلال أسبوع |
| اختبار شامل | المطور | خلال أسبوع |

### 7.2 التحقق من نجاح الإصلاح

```
1. اختبار الثغرة المعالجة
2. اختبار الأداء بعد الإصلاح
3. اختبار واجهة المستخدم
4. مراقبة لمدة 48 ساعة بعد الإصلاح
5. تأكيد عدم تكرار الحادث
```

## 8. مرحلة الإبلاغ — Reporting Phase (CNPDP Notification)

### 8.1 الإبلاغ لمكتب حماية البيانات الشخصية (CNPDP)

| العنصر | التفاصيل |
|--------|----------|
| **المدة** | خلال 72 ساعة من اكتشاف الحادث (المادة 43 من القانون 18-07) |
| **المحتوى** | وصف الحادث، النطاق، الإجراءات المتخذة |
| **الجهة** | Commission Nationale de Protection des Données Personnelles |
| **الطريقة** | إلكتروني + ورقي |

### 8.2 محتوى الإبلاغ

يجب أن يتضمن الإبلاغ:

```
1. وصف الحادث
2. فئة البيانات المتضررة
3. عدد الأشخاص المتضررين
4. الآثار المحتملة
5. الإجراءات المتخذة للإصلاح
6. الاتصال بالمسؤول المعني
7. التقييم للمخاطر المتبقية
```

### 8.3 متى لا يُبلَّغ؟

- إذا لم تتأثر بيانات شخصية
- إذا كان الحادث محدود النطاق ولا يُشكل تهديداً للحقوق
- إذا اتخذت إجراءات فعالة تمنع الآثار السلبية

> **ملاحظة:** في حالة الشك، يُبلَّغ دائماً. الإبلاغ المتأخر أفضل من عدم الإبلاغ.

## 9. مرحلة إشعار المعنيين — Data Subject Notification

### 9.1 متى يُشَكَّر المستخدمون؟

يُشَكَّر المستخدمون المتضررون في الحالات التالية:
- تسريب بيانات شخصية
- خطر كبير على الحقوق والحريات
- بيانات حساسة متضمنة (blood type، health-related)

### 9.2 محتوى الإشعار

```
1. وصف الحادث بلغة واضحة
2. نوع البيانات المتضررة
3. الإجراءات المتخذة
4. نصائح للمستخدمين (تغيير كلمة المرور)
5. معلومات الاتصال للاستفسارات
6. حقهم في تقديم شكوى لدى CNPDP
```

### 9.3 طرق الإشعار

| الطريقة | الأولوية |
|---------|---------|
| البريد الإلكتروني | الأولى |
| إشعار في التطبيق | الثانية |
| الصفحة الرئيسية | الثالثة |

## 10. مرحلة استعادة الخدمة — Service Recovery Phase

### 10.1 إجراءات الاستعادة

```
1. التأكد من إصلاح جميع الثغرات
2. استعادة الخدمة تدريجياً
3. مراقبة الأداء والأمان
4. تأكيد استقرار النظام
5. استئناف الحركة الطبيعية
```

### 10.2 معايير استعادة الخدمة

| المعيار | القيمة |
|---------|--------|
| RTO الفعلي | ≤ 4 ساعات |
| سلامة البيانات | 100% مُتحقق منها |
| اختبارات الأداء | ناجحة |
| لا ثغرات معروفة | مُتحقق |

### 10.3 المراقبة بعد الاستعادة

- مراقبة مستمرة لمدة 7 أيام
- تحليل السجلات يومياً
- مراجعة التنبيهات
- تقرير يومي لأول أسبوع

---

## 11. مرحلة ما بعد الحادث — Post-Incident Phase (Lessons Learned)

### 11.1 اجتماع ما بعد الحادث

يُعقد خلال أسبوع من الحادث ويتضمن:

```
1. مراجعة تسلسل الأحداث
2. تقييم فعالية الاستجابة
3. تحديد نقاط القوة والضعف
4. اقتراح تحسينات
5. تحديث السياسات إن لزم
6. توثيق الدروس المستفادة
```

### 12.2 تقرير ما بعد الحادث

يتضمن التقرير:

| القسم | المحتوى |
|-------|---------|
| **ملخص الحادث** | وصف موجز |
| **السبب الجذري** | السبب الأساسي للحادث |
| **النطاق** | حجم البيانات والمستخدمين المتضررين |
| **فعالية الاستجابة** | تقييم أداء الفريق |
| **التحسينات** | إجراءات وقائية مقترحة |
| **الخطة الزمنية** | جدول تنفيذ التحسينات |
| **المسؤول** | المسؤول عن كل إجراء |

### 11.3 التحسينات المحتملة

```
- تحسين المراقبة والأمان
- تحديث السياسات
- تدريب الفريق
- تحسين أدوات الاستجابة
- تحديث خطة الاستجابة نفسها
```

## 12. جدول الاستجابة الزمني — Response Timeline

| المرحلة | المدة | الإجراءات الرئيسية |
|---------|-------|-------------------|
| **الاكتشاف** | 0-1 ساعة | تسجيل الحادث، تحديد الخطورة، تشكيل الفريق |
| **الاحتواء** | 1-4 ساعات | عزل النظام، حجب المشبوه، تغيير كلمات المرور |
| **التحقيق** | 4-72 ساعة | تحليل السبب الجذري، توثيق الأدلة |
| **الإصلاح** | 24-168 ساعة | سد الثغرة، تحديثات الأمان |
| **الإبلاغ (CNPDP)** | ≤ 72 ساعة | إبلاغ CNPDP |
| **إشعار المعنيين** | ≤ 72 ساعة | إشعار المستخدمين المتضررين |
| **استعادة الخدمة** | 4-24 ساعة | استعادة تدريجية، مراقبة |
| **ما بعد الحادث** | 1-4 أسابيع | تقرير، تحسينات، تحديث السياسات |
