# تدفق البيانات — Data Flow Diagram

**المشروع:** Qatra - منصة التبرع بالدم في الجزائر
**الإصدار:** 1.0.0
**آخر تحديث:** 2026-07-13

---

## ملخص البنية التحتية

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   المستخدم   │ ──▶ │   المتصفح   │ ──▶ │   Supabase  │ ──▶ │ PostgreSQL  │
│  (Browser)   │     │  (CDN/API)  │     │   (Edge +   │     │  (Database) │
│              │     │             │     │    Auth)    │     │             │
└─────────────┘     └─────────────┘     └─────────────┘     └─────────────┘
       │                   │                   │                    │
       │                   │                   │                    │
       ▼                   ▼                   ▼                    ▼
  المدخلات:          المعالجة:           الخادم:             التخزين:
  - تسجيل            - التحقق           - المصادقة          - بيانات المتبرعين
  - بحث              - التشفير           - RLS               - طلبات الدم
  - طلب تبرع         - التحقق من         - API               - التقييمات
  - تقييم               الصلاحيات        - Edge Functions    - سجلات التدقيق
  - ملاحظة                               - Storage           - سجلات الموافقات
```

## 1. تدفق التسجيل — User Registration Flow

### 1.1 المخطط النصي

```
المستخدم ──▶ إدخال البيانات ──▶ التحقق من صحة البيانات ──▶ إنشاء الحساب ──▶ إرسال OTP
     │              │                    │                      │               │
     ▼              ▼                    ▼                      ▼               ▼
  المدخلات:    التحقق من:          Supabase Auth:         إنشاء سجل:     إرسال رسالة:
  - الاسم      - صيغة البريد       - إنشاء مستخدم        - جدول donors  - OTP إلى
  - البريد     - صحة الهاتف        - تشفير كلمة المرور   - بيانات        البريد
  - الهاتف     - نوع الدم          - إنشاء جلسة            شخصية        أو الهاتف
  - نوع الدم   - الولاية                                - مسجلة
  - الولاية    - Commune                                - timestamp
  - Commune
  - الموافقة
```

### 1.2 التفاصيل

| المرحلة | الإدخال | المعالجة | التخزين | الإخراج |
|---------|--------|---------|---------|---------|
| **1. إدخال البيانات** | بيانات المستخدم + موافقة | التحقق من صحة البيانات | لا شيء بعد | خطأ أو متابعة |
| **2. إنشاء الحساب** | البيانات الصالحة | Supabase Auth: إنشاء مستخدم + تشفير | `auth.users` | معرّف المستخدم |
| **3. إنشاء الملف الشخصي** | معرّف المستخدم | إدراج في جدول المتبرعين | `donors` | ملف شخصي |
| **4. إرسال OTP** | البريد/الهاتف | إنشاء OTP + إرسال | `auth.mfa_factors` | OTP للمستخدم |
| **5. التحقق** | OTP | التحقق من OTP | `donors.verified = true` | حساب مُفعّل |

### 1.3 من يصل إلى ماذا؟

| الجهة | الوصول |
|-------|--------|
| **المستخدم** | بياناته الشخصية فقط |
| **Supabase Auth** | بيانات المصادقة (auth.users) |
| **PostgreSQL (RLS)** | فقط بيانات المستخدم (donors) |
| **المطور** | لا يصل للبيانات الشخصية (RLS) |

## 2. تدفق بحث المتبرعين — Donor Search Flow

### 2.1 المخطط النصي

```
المستخدم ──▶ اختيار المعايير ──▶ إرسال الاستعلام ──▶ Supabase Edge Function
     │              │                    │                    │
     ▼              ▼                    ▼                    ▼
  المدخلات:    معايير البحث:         API Request:        معالجة:
  - نوع الدم   - نوع الدم المطلوب    - GET /donors       - تطبيق RLS
  - الولاية    - الولاية             - query params       - فلترة النتائج
  - Commune    - Commune             - Authentication     - إرجاع البيانات
                               │
                               ▼
                     PostgreSQL:
                     - RLS policies
                     - WHERE clauses
                     - نتائج مفلترة
                               │
                               ▼
                     للمستخدم:
                     - قائمة المتبرعين
                     - بيانات اتصال محدودة
                     - (بدون أرقام الهاتف في الاستعلام)
```

### 2.2 التفاصيل

| المرحلة | الإدخال | المعالجة | التخزين | الإخراج |
|---------|--------|---------|---------|---------|
| **1. تحديد المعايير** | نوع الدم، الولاية، Commune | لا شيء | لا شيء | معايير البحث |
| **2. إرسال الاستعلام** | معايير البحث | Edge Function: تطبيق RLS | لا شيء | استعلام PostgreSQL |
| **3. معالجة RLS** | الاستعلام | التحقق من الصلاحيات + الفلترة | لا شيء | نتائج مفلترة |
| **4. عرض النتائج** | النتائج | تنسيق البيانات | لا شيء | قائمة المتبرعين |

### 2.3 من يصل إلى ماذا؟

| الجهة | الوصول |
|-------|--------|
| **المستخدم الباحث** | اسم المتبرع + نوع الدم + الولاية + Commune |
| **المستخدم (صاحب الملف)** | بياناته الشخصية الكاملة فقط |
| **Supabase** | الاستعلام فقط (لا يخزّن معايير البحث) |
| **PostgreSQL** | جميع البيانات (محفوظة مسبقاً) |

### 2.4 ملاحظات أمنية

- لا تُعرض أرقام الهواتف في نتائج البحث العامة
- تتطلب المصادقة للبحث
- RLS تمنع الوصول غير المصرح به

## 3. تدفق الاتصال (مكالمة/WhatsApp) — Contact Flow

### 3.1 المخطط النصي

```
المستخدم الباحث ──▶ اختيار متبرع ──▶ طلب الاتصال ──▶ تسجيل الطلب
       │                  │                  │                │
       ▼                  ▼                  ▼                ▼
  المدخلات:         اختيار المتبرع:      Edge Function:    PostgreSQL:
  - المتبرع         - المعرّف           - التحقق من       - تسجيل في
    المُختار        - السبب               الصلاحيات        blood_requests
                                       - إنشاء سجل
                                       - إرجاع معلومات
                                           الاتصال
       │
       ▼
  للمستخدم الباحث:
  - رقم هاتف المتبرع (إذا وافق)
  - أو: رسالة "تم إرسال طلب"
       │
       ▼
  للمتبرع:
  - إشعار بطلب اتصال
  - يمكنه القبول أو الرفض
```

### 3.2 التفاصيل

| المرحلة | الإدخال | المعالجة | التخزين | الإخراج |
|---------|--------|---------|---------|---------|
| **1. اختيار المتبرع** | معرّف المتبرع | لا شيء | لا شيء | لا شيء |
| **2. طلب الاتصال** | معرّف المتبرع + السبب | Edge Function: التحقق + إنشاء سجل | `blood_requests` | طلب مسجل |
| **3. إشعار المتبرع** | معرّف الطلب | إرسال إشعار | لا شيء | إشعار للمتبرع |
| **4. قبول/رفض** | قرار المتبرع | تحديث حالة الطلب | `blood_requests.status` | معلومات الاتصال أو رفض |

### 3.3 من يصل إلى ماذا؟

| الجهة | الوصول |
|-------|--------|
| **المستخدم الباحث** | رقم الهاتف فقط بعد قبول المتبرع |
| **المتبرع** | بيانات الطلب (نوع الدم، السبب) |
| **Supabase** | لا يصل للهاتف (يُخزَّن مؤقتاً فقط) |

## 4. تدفق تقديم التقييم — Rating Submission Flow

### 4.1 المخطط النصي

```
المستخدم ──▶ اختيار المتبرع ──▶ إدخال التقييم ──▶ التحقق من الأهلية ──▶ حفظ التقييم
     │              │                  │                    │                │
     ▼              ▼                  ▼                    ▼                ▼
  المدخلات:    المتبرع:             التقييم:           التحقق:           PostgreSQL:
  - المتبرع    - المعرّف            - النجوم (1-5)    - هل توجد          - INSERT INTO
  - النجوم     - السبب (اختياري)    - التعليق          اتصال سابق؟       ratings
  - التعليق                                             - هل قيّم سابقاً؟
```

### 4.2 التفاصيل

| المرحلة | الإدخال | المعالجة | التخزين | الإخراج |
|---------|--------|---------|---------|---------|
| **1. اختيار المتبرع** | معرّف المتبرع | لا شيء | لا شيء | لا شيء |
| **2. إدخال التقييم** | النجوم + التعليق | التحقق من الأهلية | لا شيء | تقييم صالح |
| **3. الحفظ** | التقييم | Edge Function: INSERT | `ratings` | تقييم مسجل |

## 5. تدفق تقديم الملاحظات — Feedback Submission Flow

### 5.1 المخطط النصي

```
المستخدم ──▶ إدخال الملاحظة ──▶ اختيار التصنيف ──▶ الإرسال ──▶ الحفظ
     │              │                  │               │            │
     ▼              ▼                  ▼               ▼            ▼
  المدخلات:    نص الملاحظة:       التصنيف:         Edge Function:  PostgreSQL:
  - النص       - المحتوى          -bug report     - التحقق       - INSERT INTO
  - التصنيف    - الطول ≤ 2000     - feature req    - التنظيف       feedback
                             - general
```

### 5.2 التفاصيل

| المرحلة | الإدخال | المعالجة | التخزين | الإخراج |
|---------|--------|---------|---------|---------|
| **1. إدخال الملاحظة** | النص + التصنيف | تنظيف المدخلات | لا شيء | ملاحظة نظيفة |
| **2. الإرسال** | الملاحظة النظيفة | Edge Function: INSERT | `feedback` | ملاحظة مسجلة |

## 6. تدفق حذف البيانات — Data Deletion Flow

### 6.1 المخطط النصي

```
المستخدم ──▶ طلب الحذف ──▶ التحقق ──▶ حذف الحساب ──▶ حذف البيانات ──▶ تأكيد
     │              │            │              │                │            │
     ▼              ▼            ▼              ▼                ▼            ▼
  الإدخال:      Edge Function: Supabase Auth:  PostgreSQL:     النسخ:
  - تأكيد       - التحقق من    - حذف المستخدم  - حذف جميع     - تسجيل في
    الحذف         هوية المستخدم  من auth.users    السجلات        audit_logs
               - التحقق من                      المرتبطة
                 كلمة المرور                     (CASCADE)
```

### 6.2 التفاصيل

| المرحلة | الإدخال | المعالجة | التخزين | الإخراج |
|---------|--------|---------|---------|---------|
| **1. طلب الحذف** | تأكيد المستخدم | Edge Function: التحقق | لا شيء | طلب صالح |
| **2. حذف الحساب** | معرّف المستخدم | Supabase Auth: حذف | `auth.users` (محذوف) | لا شيء |
| **3. حذف البيانات** | معرّف المستخدم | CASCADE delete | `donors` (محذوف) | لا شيء |
| **4. التسجيل** | تفاصيل الحذف | INSERT | `audit_logs` | سجل الحذف |
| **5. التأكيد** | لا شيء | لا شيء | لا شيء | رسالة تأكيد |

### 6.3 البيانات المحذوفة

```
✅ donors (المعلومات الشخصية)
✅ blood_requests (طلبات الدم المرتبطة)
✅ ratings (التقييمات التي قدمها أو تلقاها)
✅ feedback (الملاحظات)
✅ consent_records (سجلات الموافقة — بعد 5 سنوات)
❌ audit_logs (تبقى لغرض التدقيق — تُحذف بعد 3 سنوات)
❌ wilayas/communes (بيانات مرجعية — لا تُحذف)
```

## 7. تدفق الوصول إلى البيانات — Data Access Flow

### 7.1 المخطط النصي

```
المستخدم ──▶ طلب الوصول ──▶ التحقق ──▶ جمع البيانات ──▶ عرض البيانات
     │              │            │              │                │
     ▼              ▼            ▼              ▼                ▼
  الإدخال:      Edge Function: Supabase Auth:  PostgreSQL:     للمستخدم:
  - تسجيل       - التحقق من    - التحقق من     - SELECT        - بياناته
    الدخول         هوية المستخدم  الصلاحيات      الشخصية       الشخصية
                                     │         - RLS filter   - سجلات
                                     ▼                        المعالجة
                                لا يمكن الوصول
                                لبيانات المستخدمين
                                الآخرين (RLS)
```

### 7.2 مصفوفة الوصول

| الجهة | donors | blood_requests | ratings | feedback | consent_records | audit_logs | wilayas |
|-------|--------|----------------|---------|----------|-----------------|------------|---------|
| **المستخدم (صاحب البيانات)** | قراءة/كتابة | قراءة/كتابة | قراءة/كتابة | قراءة/كتابة | قراءة | قراءة | قراءة |
| **المستخدم (بيانات الآخرين)** | محدود | محدود | محدود | لا | لا | لا | قراءة |
| **المطور** | لا (RLS) | لا (RLS) | لا (RLS) | لا (RLS) | لا (RLS) | لا (RLS) | قراءة |
| **Supabase** | إداري | إداري | إداري | إداري | إداري | إداري | إداري |
| **CNPDP** | طلب قانوني | طلب قانوني | طلب قانوني | طلب قانوني | طلب قانوني | طلب قانوني | لا |

### 7.3 ملاحظات أمنية

- **RLS (Row Level Security):** يمنع الوصول غير المصرح به على مستوى قاعدة البيانات
- **MFA (Multi-Factor Authentication):** يُنفَّذ لاحقاً لحماية إضافية
- **Audit Logging:** يُسجَّل كل وصول للبيانات
- **Encryption:** جميع البيانات مشفرة أثناء الراحة والنقل

## 8. ملخص تدفق البيانات

### 8.1 نقاط الدخول

| نقطة الدخول | البيانات المُجمَّعة | الأساس القانوني |
|-------------|-------------------|----------------|
| التسجيل | الاسم، البريد، الهاتف، نوع الدم، الولاية، Commune | الموافقة |
| طلب الاتصال | نوع الدم المطلوب، السبب | تنفيذ الخدمة |
| التقييم | النجوم، التعليق | المصلحة المشروعة |
| الملاحظة | النص، التصنيف | الموافقة |

### 8.2 نقاط الخروج

| نقطة الخروج | البيانات المُصدَّرة | المتلقي |
|-------------|-------------------|---------|
| بحث المتبرعين | الاسم، نوع الدم، الولاية | المستخدم الباحث |
| طلب الاتصال | رقم الهاتف (بعد القبول) | المستخدم الباحث |
| تصدير البيانات | جميع البيانات الشخصية | صاحب البيانات |
| حذف الحساب | لا شيء (تدمير) | لا أحد |

### 8.3 نقاط التخزين

| نقطة التخزين | البيانات | المدة | التشفير |
|-------------|---------|-------|---------|
| PostgreSQL (donors) | بيانات المتبرعين | حتى طلب الحذف أو سنتين من عدم النشاط | AES-256 |
| PostgreSQL (blood_requests) | طلبات الدم | 90 يوماً | AES-256 |
| PostgreSQL (ratings) | التقييمات | سنتان | AES-256 |
| PostgreSQL (feedback) | الملاحظات | سنة | AES-256 |
| PostgreSQL (consent_records) | سجلات الموافقة | 5 سنوات | AES-256 |
| PostgreSQL (audit_logs) | سجلات التدقيق | 3 سنوات | AES-256 |
| Supabase Auth | بيانات المصادقة | حتى حذف الحساب | AES-256 |
