# سياسة الاحتفاظ بالبيانات — Data Retention Policy

**المشروع:** ق traced - منصة التبرع بالدم في الجزائر
**الإصدار:** 1.0.0
**آخر تحديث:** 2026-07-13
**المسؤول:** مسؤول حماية البيانات (DPO)

---

## 1. المقدمة

تُحدد هذه السياسة مدد الاحتفاظ بالبيانات الشخصية التي تجمعها منصة "ق traced" (Qatra)، والطرق المستخدمة في حذفها أو إتلافها، وذلك امتثالاً ل:

- القانون 18-07 المتعلق بحماية الأشخاص الطبيعيين عند معالجة بياناتهم الشخصية
- اللائحة العامة لحماية البيانات (GDPR) كمرجع إضافي
- الممارسات الأمثل المعتمدة في هندسة الخصوصية (Privacy by Design)

---

## 2. الأطراف المسؤولة

| الدور | المسؤولية |
|-------|----------|
| مسؤول حماية البيانات (DPO) | الإشراف العام على الامتثال لسياسة الاحتفاظ |
| المطور (Developer) | تنفيذ عمليات الحذف التقنية والآلي |
| المستخدم (Data Subject) | ممارسة حقه في الحذف عبر واجهة المنصة |

## 3. جدول الاحتفاظ بالبيانات حسب الجدول

### 3.1 جدول المتبرعين (donors)

| العنصر | التفاصيل |
|--------|----------|
| **البيانات** | الاسم، البريد الإلكتروني، رقم الهاتف،啵 blood type، الولاية، Commune، حالة النشاط، تاريخ التسجيل، رموز التحقق |
| **مدة الاحتفاظ** | حتى يطلب المستخدم الحذف، أو سنتان من عدم النشاط (88 يوماً+) |
| **طريقة الحذف** | حذف ذاتي عبر الملف الشخصي، أو حذف آلي بعد سنتين من عدم النشاط |
| **سبب المدة** | الاحتفاظ بالبيانات ضروري لتقديم الخدمة؛ يحق للمستخدم طلب الحذف في أي وقت بموجب المادة 26 من القانون 18-07 |
| **البيانات المتبقية بعد الحذف** | لا تُحتفظ بأي بيانات شخصية بعد الحذف |

### 3.2 جدول طلبات الدم (blood_requests)

| العنصر | التفاصيل |
|--------|----------|
| **البيانات** | نوع الدم المطلوب، الولاية، المستشفى/المؤسسة، حالة الطلب، التاريخ، معلومات مرتبطة بالطلب |
| **مدة الاحتفاظ** | 90 يوماً من تاريخ الإنشاء |
| **طريقة الحذف** | حذف آلي دوري (يومياً) عبر cron job يحذف الطلبات التي تجاوزت 90 يوماً |
| **سبب المدة** | الطلبات ت失去 صلاحيتها بعد 90 يوماً؛ الحفاظ على أداء قاعدة البيانات |
| **الملاحظات** | لا تتضمن هذه الطلبات بيانات شخصية حساسة |

### 3.3 جدول التقييمات (ratings)

| العنصر | التفاصيل |
|--------|----------|
| **البيانات** | التقييم (1-5 نجوم)، المعلق، المستخدم المُقيّم، المستخدم المُقيّم، التاريخ |
| **مدة الاحتفاظ** | سنتان من تاريخ الإنشاء |
| **طريقة الحذف** | حذف آلي بعد سنتين |
| **سبب المدة** | الحفاظ على سجل التقييمات لأغراض الجودة ثم الحذف تلقائياً |

### 3.4 جدول الملاحظات والاقتراحات (feedback)

| العنصر | التفاصيل |
|--------|----------|
| **البيانات** | نص الملاحظة، التصنيف، تاريخ الإرسال، المستخدم (اختياري) |
| **مدة الاحتفاظ** | سنة واحدة من تاريخ الإنشاء |
| **طريقة الحذف** | حذف آلي بعد سنة |
| **سبب المدة** | تحليل جودة الخدمة ثم الحذف |

### 3.5 جداول المرجعية (wilayas, communes, blood_types)

| العنصر | التفاصيل |
|--------|----------|
| **البيانات** | أسماء الولايات، البلديات، أنواع الدم، الرموز البريدية |
| **مدة الاحتفاظ** | دائمة (لا يُحذف) |
| **طريقة الحذف** | غير قابل للحذف — بيانات مرجعية |
| **سبب المدة** | بيانات مرجعية ضرورية لعمل المنصة؛ لا تتضمن بيانات شخصية |

### 3.6 جدول سجلات الموافقات (consent_records)

| العنصر | التفاصيل |
|--------|----------|
| **البيانات** | نوع الموافقة، تاريخ الموافقة، تاريخ سحب الموافقة، IP Address، User Agent |
| **مدة الاحتفاظ** | 5 سنوات من تاريخ آخر موافقة (أو سحب موافقة) |
| **طريقة الحذف** | حذف آلي بعد 5 سنوات |
| **سبب المدة** | المتطلبات القانونية بموجب القانون 18-07 — إثبات الموافقة |

### 3.7 جدول سجلات التدقيق (audit_logs)

| العنصر | التفاصيل |
|--------|----------|
| **البيانات** | العملية، المستخدم المعني، التاريخ والوقت، عنوان IP، التفاصيل التقنية |
| **مدة الاحتفاظ** | 3 سنوات |
| **طريقة الحذف** | حذف آلي بعد 3 سنوات |
| **سبب المدة** | التدقيق الداخلي، اكتشاف الأخطاء، الامتثال القانوني |

## 4. جدول الحذف الآلي — Auto-Deletion Schedule

```
يومياً (02:00 UTC):
  - حذف طلبات الدم (blood_requests) التي تجاوزت 90 يوماً

أسبوعياً (الأحد 03:00 UTC):
  - فحص المتبرعين غير النشطين (سنتان+) وتحديد الحذف
  - حذف سجلات الموافقات التي تجاوزت 5 سنوات

شهرياً (الأول من كل شهر 04:00 UTC):
  - حذف التقييمات التي تجاوزت سنتين
  - حذف الملاحظات التي تجاوزت سنة
  - حذف سجلات التدقيق التي تجاوزت 3 سنوات
  - إنشاء تقرير الحذف الشهري
```

### تنفيذ تقني

```sql
-- حذف طلبات الدم القديمة (يومياً)
DELETE FROM blood_requests
WHERE created_at < NOW() - INTERVAL '90 days';

-- حذف المتبرعين غير النشطين (بعد تنبيه مسبق)
UPDATE donors
SET status = 'inactive'
WHERE last_active_at < NOW() - INTERVAL '18 months'
  AND status = 'active';

-- حذف المتبرعين غير النشطين بعد سنتين
DELETE FROM donors
WHERE last_active_at < NOW() - INTERVAL '2 years'
  AND status = 'inactive';

-- حذف التقييمات القديمة
DELETE FROM ratings
WHERE created_at < NOW() - INTERVAL '2 years';

-- حذف الملاحظات القديمة
DELETE FROM feedback
WHERE created_at < NOW() - INTERVAL '1 year';

-- حذف سجلات الموافقات القديمة
DELETE FROM consent_records
WHERE updated_at < NOW() - INTERVAL '5 years';

-- حذف سجلات التدقيق القديمة
DELETE FROM audit_logs
WHERE created_at < NOW() - INTERVAL '3 years';
```

## 5. عملية الحذف اليدوي — Manual Deletion Process

### 5.1 طلب المستخدم للحذف

1. ي-submit المستخدم طلب حذف عبر صفحة الملف الشخصي أو البريد الإلكتروني
2. تتم مراجعة الطلب من قبل DPO خلال 48 ساعة
3. يُنفَّذ الحذف خلال مدة أقصاها 30 يوماً
4. يُرسل تأكيد للمستخدم

### 5.2 حذف DPO

يمكن لـ DPO حذف بيانات مستخدم محدد في الحالات التالية:
- طلب قانوني من الجهات المختصة
- اكتشاف بيانات مزيفة أو مخالفة
- خرق أمني

### 5.3 الإجراءات التقنية

```
1. تعطيل الحساب (soft delete) — فوراً
2. إزالة البيانات الشخصية (hard delete) — خلال 30 يوماً
3. تنظيف النسخ الاحتياطية — خلال دورة النسخ الاحتياطي التالية
4. التحقق من الحذف — تقرير موثق
```

---

## 6. حذف النسخ الاحتياطية — Backup Deletion

### 6.1 النسخ الاحتياطية على Supabase

- تُدار النسخ الاحتياطية تلقائياً من قبل Supabase
- تُحذف النسخ الأقدم من 30 يوماً تلقائياً
- لا يمكن حذف نسخة احتياطية فردية يدوياً

### 6.2 النسخ الاحتياطية اليدوية (SQL exports)

- تُخزَّن في مخزن آمن مشفر
- تُحذف بعد 90 يوماً من الإنشاء
- تُنظَّف تلقائياً عبر سكربت

### 6.3 حذف النسخ الاحتياطية بعد حذف المستخدم

عند حذف بيانات مستخدم:
1. يتم تحديد السجلات المرتبطة في النسخة الاحتياطية الحالية
2. تُعلَّم السجلات للحذف في الدورة التالية
3. لا يمكن ضمان الحذف الفوري من النسخ الاحتياطية

> **ملاحظة مهمة:** النسخ الاحتياطية المشفرة على Supabase تُحذف تلقائياً بعد 30 يوماً. هذا يعني أن البيانات المحذوفة ستختفي من النسخ الاحتياطية خلال 30 يوماً كحد أقصى.

## 7. استثناء الإجراء القانوني — Legal Hold Exceptions

### 7.1 ما هو الإجراء القانوني (Legal Hold)؟

إشعار قانوني يُصدِرته جهة مختصة يطلب الاحتفاظ ببيانات معينة على الرغم من سياسات الحذف.

### 7.2 الإجراءات عند تلقي إشعار قانوني

1. **التوثيق:** تسجيل الإشعار في سجل التدقيق
2. **تعليق الحذف:** إيقاف الحذف الآلي للبيانات المعنية
3. **الإشعار:** إخطار DPO والفريق التقني
4. **المراقبة:** مراقبة مدة الإشعار القانوني
5. **التنفيذ:** تنفيذ الحذف فور انتهاء الإشعار القانوني

### 7.3 البيانات الخاضعة للاستثناء

- سجلات التدقيق (audit_logs) — قد تُطلب في تحقيقات أمنية
- سجلات الموافقات (consent_records) — إثبات الامتثال
- أي بيانات تطلبها جهة قضائية مختصة

## 8. تنبيه عدم النشاط — Inactivity Warning

### 8.1 مراحل التنبيه

| المدة | الإجراء |
|-------|---------|
| 12 شهراً | تنبيه بالبريد الإلكتروني: "لم نلاحظ نشاطك منذ فترة" |
| 18 شهراً | تنبيه ثاني: "حسابك سيُحذف خلال 6 أشهر إذا لم تستخدم المنصة" |
| 24 شهراً | حذف الحساب والبيانات المرتبطة |

### 8.2 إعادة التنشيط

- يمكن للمستخدم إعادة تنشيط حسابه خلال فترة التنبيه
- إعادة التنشيط تُلغي عداد عدم النشاط

## 9. توثيق وتدقيق — Documentation & Audit

### 9.1 السجلات المطلوب الاحتفاظ بها

- سجل جميع عمليات الحذف (التاريخ، المستخدم، السبب)
- سجل تقارير الحذف الشهري
- سجل طلبات حذف المستخدمين
- سجل الإجراءات القانونية (Legal Holds)

### 9.2 التقارير الدورية

- **تقرير شهري:** ملخص عمليات الحذف الآلي
- **تقرير ربع سنوي:** مراجعة شاملة لسياسات الاحتفاظ
- **تقرير سنوي:** مراجعة وتحديث السياسة
